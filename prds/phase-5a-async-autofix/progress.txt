## Codebase Patterns
- DRCRunner has both sync `run()` and async `async_run()` methods — tests can use either
- API tests use `TestClient(app)` with patched `cfg.JOBS_DIR` and `cfg.UPLOAD_DIR` for isolation
- The DRC route now returns immediately; background task updates job status via `manager.update_status()`
- Mock async subprocess with `AsyncMock` from `unittest.mock`, but use `MagicMock` for `proc.kill()` (non-coroutine)
- `pytest-asyncio` with `asyncio_mode = "auto"` — no need for `@pytest.mark.asyncio` decorator
- AutoFixRunner is a standalone service class (backend/fix/autofix.py) — separate from route handlers
- When testing auto-fix loop, patch `_apply_deltas_from_suggestions` since LayoutManager is mocked
- `_is_auto_applicable()` and `_flag_reason()` are module-level functions, easy to unit test directly
- Database creates both `jobs` and `fix_provenance` tables in `__init__` — no separate migration needed
- `_write_provenance()` in autofix.py writes one row per PolygonDelta (not per suggestion)
- JSON-serialized points come back as lists-of-lists, not lists-of-tuples — test accordingly
- `JobManager` exposes `insert_provenance()`, `get_provenance()`, `update_provenance_action()` delegating to Database
- `_detect_oscillation()` and `_extract_category_counts()` are pure module-level functions — unit test directly
- Category history is seeded from the initial report (iteration 0), then updated after each DRC re-run

## 2026-03-03 - P5-AF-001

- Implements loop: suggest → filter by confidence → apply high-confidence fixes → re-DRC → repeat
